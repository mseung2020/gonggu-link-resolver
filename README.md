# 공구왕 링크 해석 모듈 (프로토타입)

인스타그램/유튜브 공동구매(공구) 포스트 하나를 입력받아, 실제 판매 페이지(상품명·가격·이미지)를 찾아내거나
"왜 못 찾았는지"까지 분류해서 결론을 내는 파이프라인입니다. 사람이 직접 링크를 따라가며 확인하던 과정을
LLM 판단(저비용 모델 3개) + 크롤링 스크립트로 자동화합니다.

## 왜 이렇게 설계했는가

공구 포스트는 캡션에 구매 링크가 바로 있는 경우가 드뭅니다 (인스타그램 기준 약 1%). 대부분은 "프로필 링크
확인"으로 유도하고, 프로필 링크도 대개 여러 상품을 모아둔 "링크모음" 페이지라 한 번 더 들어가서 맞는 링크를
골라야 합니다. 이 과정에서 실제로 겪은 문제들:

- 캡션의 "오늘"이 작성 시점 기준이라, LLM이 캡션 문구만 보고 진행상태를 판단하면 날짜 계산이 틀립니다 →
  **명시적 날짜가 있으면 코드가 실제 오늘 날짜와 직접 비교해서 계산**하고, LLM 판단은 힌트로만 씁니다.
- 링크모음 페이지에서 아무 링크나 고르면 다른 상품이 잡힙니다 → **원본 포스트의 상품과 실제로 일치하는지를
  별도 LLM 단계가 명시적으로 검증**합니다.
- 크리에이터가 원본부터 잘려서 캡션에 올린 링크, 스킴 오타(`https//...`), 느린 리다이렉트(`mkt.shopping.naver.com`)
  등 크롤링 단계의 실제 노이즈들을 하나씩 겪고 고쳤습니다.

## 아키텍처

```
포스트(캡션+발행일+프로필bio)
   │
   ▼
[LLM #1] 링크방식분류  ─ 진짜 공구인지 / 상품 요약 / 명시적 시작·종료일 / 진행상태 힌트 / 링크 위치·목록
   │
   ▼
[게이트, 코드]  is_gonggu=false / 날짜 계산상 예고·마감 / 링크 전무  → 종료(원천 불가)
   │
   ▼
[링크 준비, 코드]  제휴광고성 다중링크 스킵 / 잘린 링크 건너뛰기 / URL 오타 보정
   │
   ▼
┌─ [크롤러, 코드] 페이지 방문 → 제목/og:image/JSON-LD/본문텍스트 수집 (느린 리다이렉트·지연 렌더링 재시도)
│     │
│     ▼
│  [LLM #3] 페이지판별 ─ 상품페이지/스토어메인/로그인월_차단/링크모음/무관 + 원본 상품과 실제 일치하는지
│     │
│     ├─ 상품페이지 + 일치 ────────────────────────────► ✅ 완료
│     ├─ 링크모음 ─┐
│     │            ▼
│     │        [크롤러] 후보 링크 추출(최대 80개)
│     │            ▼
│     │        [LLM #2] 링크선택 ─ 후보 중 원본 상품과 맞는 것 고르기
│     │            │
│     └────────────┘  (최대 3홉까지 위로 되돌아감)
│
└─ 스토어메인 / 로그인월_차단 / 무관 / 상품 불일치 / 최대홉초과 ──► 미해결(사유별로 구분)
```

자세한 그림은 `dashboard/pipeline_diagram.html`을 브라우저로 열어서 보세요.

## 디렉토리 구조

```
dify_workflows/       Dify DSL yml 3개 — Dify에 "DSL로 가져오기"로 import
  01_link_classification.yml    LLM #1
  02_link_selection.yml        LLM #2
  03_page_judge.yml      LLM #3
scripts/
  resolver.py            핵심 오케스트레이터 (게이트 → 크롤 → LLM#2/#3 홉 루프)
  test_run_store.py      진행 상태 저장소 (JSON 파일, resolver가 매 홉마다 기록)
  dashboard_server.py    로컬 대시보드 서버 (:8010)
  run_dify_batch.py      샘플 전체에 LLM #1을 배치로 미리 돌려두는 스크립트
  prep_dify_input.py     플랫폼별 필드명을 LLM #1 입력 스키마에 맞게 정규화
  build_creator_bio_ig.py  인스타 크리에이터 bio + 프로필 링크 join
  merge_classification.py LLM #1 배치 결과를 샘플 파일에 병합
  sample_recent_100.py   원본 쿼리 결과에서 무작위 샘플 추출
  export_success_cases.py 성공(done) 케이스만 별도 파일로 추출
  backfill_store_fields.py 저장소 스키마에 필드 추가 시 기존 진행상태 보존하며 채우기
  report_stats.py        진행 상태 누적 통계
dashboard/
  case_test.html          케이스별 실시간 진행 대시보드
  pipeline_diagram.html   위 아키텍처 그림
queries/                 원본 데이터 소스 SQL (MySQL)
data/results/success_cases.json   성공 케이스 18건 (입력 캡션 + 각 홉 로그 + 최종 상품 정보)
media/                   성공 케이스의 상품 이미지
```

## 설치

```bash
pip install -r requirements.txt
playwright install chromium
cp .env.example .env   # 값 채우고 source 하거나 export
```

## Dify 워크플로우 준비

`dify_workflows/`의 3개 yml을 Dify에서 각각 "DSL로 가져오기"로 새 앱으로 import하고, 발급된 API 키를
`.env`의 `DIFY_KEY`(01), `DIFY_KEY_PICK`(02), `DIFY_KEY_JUDGE`(03)에 채웁니다. 모델은 gpt-5-mini 기준으로
프롬프트를 짰습니다(저비용 모델 의도).

## 사용법

1. **원본 데이터 확보** — `queries/`의 SQL로 최근 N일 공구 키워드 매칭 포스트를 뽑아 JSON으로 export
   (CSV는 캡션에 쉼표/줄바꿈이 있어 컬럼이 깨질 수 있어 피할 것).
2. **샘플링** — `sample_recent_100.py`로 무작위 샘플 추출. 인스타그램은 `build_creator_bio_ig.py`로
   크리에이터 bio + 프로필 링크까지 조인 (프로필 경유 케이스의 url_type 판별 정확도에 필수).
3. **LLM #1 배치 실행** — `prep_dify_input.py`로 필드 정규화 후 `run_dify_batch.py`로 분류 배치 실행,
   `merge_classification.py`로 결과 병합.
4. **리졸버 실행** — `resolver.py all 20` (플랫폼당 20건씩, 이미 처리한 건 자동 skip). 특정 건만 재시도하려면
   `resolver.py retry <id1,id2,...>`.
5. **대시보드로 확인** — `python3 dashboard_server.py` → `http://localhost:8010/` (실시간 진행상황),
   `/pipeline_diagram.html` (구조도).
6. **통계/추출** — `report_stats.py`(누적 통계), `export_success_cases.py`(성공 케이스만 별도 저장).

## 현재까지 결과 (샘플 200건 중 131건 처리)

| 결과 | 건수 |
|---|---|
| ✅ 완료(상품명·가격·이미지 확보) | 18 |
| 원천 불가 (공구 아님/예고/마감/제휴광고성) | 66 |
| 로그인월·차단 (네이버·카카오 안티봇) | 9 |
| 스토어메인(특정 상품 못 짚음) | 16 |
| 상품페이지지만 원본과 불일치 | 7 |
| 링크 선택/추출 실패 | 10 |

실제로 시도해볼 만한 케이스(원천 불가 제외) 기준 성공률 약 25~28%.

## 알려진 한계 / 다음 과제

- **네이버·카카오 안티봇**: 헤드리스 브라우저를 로그인월로 튕겨냄. 공식 API 연동이 유일한 우회로.
- **스토어메인 → 특정 상품 찾기**: 사이트 내부 검색이 필요한 가장 어려운 케이스, 아직 미구현.
- **원본 캡션 자체의 데이터 손실**: 크리에이터가 이미 잘린 링크를 캡션에 올린 경우는 복구 불가.
- 인스타그램의 "링크모음" 후보가 매우 많은 계정(100+ 링크)에서는 LLM #2 정확도가 상대적으로 낮아질 수 있음.
